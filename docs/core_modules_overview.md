# Core 目录模块职责说明

## 概述
`adl-backend/core/` 目录包含了智能体系统的核心逻辑模块，采用模块化设计，职责清晰分离。

## 主要模块分类

### 1. 推理引擎模块
**负责：智能决策的核心流程**

| 文件 | 主要职责 | 关键特点 |
|------|----------|----------|
| `reasoning_v2.py` | **新版推理管道** - 三阶段处理流程 | 模块化设计，支持插件式扩展 |
| `reasoning_v1.py` | **旧版推理引擎** - 向后兼容 | 包含LLM调用和watchdog集成 |
| `reasoning.py` | **推理基类** - 通用接口定义 | 提供基础框架 |

### 2. 提案生成模块
**负责：生成动作建议**

| 文件 | 主要职责 | 关键特点 |
|------|----------|----------|
| `proposers_v2.py` | **提案策略** - 生成动作建议 | 策略模式，支持Mock/V1切换 |
| `MockProposer` | **测试用提案器** - 确定性规则 | 用于测试和黄金验证 |
| `V1Proposer` | **生产用提案器** - 委托给v1 | 实际部署使用 |

### 3. 安全检查模块
**负责：动作安全验证和状态监控**

| 文件 | 主要职责 | 关键特点 |
|------|----------|----------|
| `guards_v2.py` | **守卫检查** - 安全验证 | 状态停滞检测，完成检测 |
| `FinishGuard` | **完成检测** - 任务完成判断 | 系统决定，不依赖LLM |
| `StateStagnationGuard` | **状态停滞检测** - 防卡死 | 滑动窗口监控状态变化 |
| `watchdog.py` | **看门狗** - 异常监控 | 规则化异常检测 |

### 4. 适配器模块
**负责：动作格式转换**

| 文件 | 主要职责 | 关键特点 |
|------|----------|----------|
| `adapters_v2.py` | **动作适配器** - 格式转换 | INTERACT→SPEAK转换 |
| `InstructAdapter` | **指令适配器** - 生成用户指令 | 模板优先，100%掌控 |

### 5. 辅助工具模块
**负责：通用功能和工具**

| 文件 | 主要职责 | 关键特点 |
|------|----------|----------|
| `common_v2.py` | **通用工具** - 共享函数 | 动作创建，工具函数 |
| `goal_evaluator.py` | **目标评估** - 任务完成判断 | 状态分析，目标检测 |
| `error_dictionary.py` | **错误字典** - 错误码管理 | 统一错误处理 |
| `step_logger.py` | **步骤日志** - 执行记录 | 调试和监控 |
| `prompt_templates.py` | **提示模板** - LLM提示词 | 系统提示词管理 |

### 6. 记忆和智能体模块
**负责：状态管理和智能体行为**

| 文件 | 主要职责 | 关键特点 |
|------|----------|----------|
| `memory.py` | **记忆系统** - 状态存储 | 向量数据库集成 |
| `agent.py` | **智能体基类** - 行为定义 | 智能体基础行为 |

## 核心工作流程

### 推理v2管道流程
```
1. FinishGuard检查 → 如果任务完成，直接返回FINISH
2. Proposer提案 → 生成动作建议（Mock或V1）
3. Guard检查 → 安全验证和状态监控
4. Adapter转换 → 根据执行模式转换动作格式
5. 返回最终动作 → 执行或指令
```

### 执行模式支持
- **ACT模式**：直接执行物理动作
- **INSTRUCT模式**：转换为用户指令（SPEAK）

## 环境变量配置

| 变量 | 作用 | 默认值 |
|------|------|--------|
| `REASONING_V2_PROPOSER` | 提案器类型 | `mock` |
| `REASONING_V2_EXECUTION_MODE` | 执行模式 | `INSTRUCT` |
| `REASONING_V2_STAGNATION_WINDOW` | 停滞检测窗口 | `4` |
| `REASONING_V2_STAGNATION_OVERRIDE` | 停滞覆盖类型 | `THINK` |

## 设计原则

### 1. 模块化设计
- 每个模块单一职责
- 清晰接口定义
- 易于测试和替换

### 2. 确定性优先
- MockProposer提供确定性输出
- FinishGuard系统决定完成
- 模板化指令生成

### 3. 可观测性
- 结构化日志输出
- 详细错误信息
- 状态监控

### 4. 向后兼容
- 保持v1接口
- 渐进式迁移
- 行为可回滚

## 使用建议

### 开发阶段
- 使用MockProposer进行确定性测试
- 启用INSTRUCT模式验证指令生成
- 利用状态停滞检测调试卡死问题

### 生产部署
- 切换到V1Proposer使用LLM推理
- 根据需求选择ACT或INSTRUCT模式
- 配置合适的停滞检测参数

### 测试验证
- 使用pipeline_test_harness进行组件测试
- 验证FinishGuard的完成检测
- 测试状态停滞检测逻辑

## 总结

Core目录采用清晰的模块化架构，每个文件都有明确的职责：
- **推理引擎**：决策流程控制
- **提案生成**：动作建议生成  
- **安全检查**：状态监控和验证
- **适配转换**：动作格式处理
- **辅助工具**：通用功能支持

这种设计使得系统易于理解、测试和维护，同时支持灵活的配置和扩展。
