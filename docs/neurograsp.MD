## 📋 项目概述

### 🎯 核心目标
开发一个以厨房日常活动为载体的严肃游戏系统，帮助自闭症个体训练日常生活技能（ADL），并通过 LLM-Agent 提供结构化解释与反馈，最终形成 **"过程数据 → 评估指标 → 个体化报告"** 的闭环，支持临床决策。

### 🧭 第一性设计原则
1. **厨房任务** 是核心干预内容（烹饪、整理、工具使用等）
2. **LLM 沟通** 是辅助解释机制，不主导任务逻辑
3. **强调可解释性、可测量性、可复用性**

### 📊 预期结局指标
- **主要指标**：任务完成率、独立步骤数、错误类型分布
- **次要指标**：家长/治疗师报告的日常技能改善

## 🔧 模块详细设计

### 1️⃣ 厨房场景层（第一性：日常任务）

#### 功能
- 执行结构化厨房任务序列（例如：煮面条 → 做三明治 → 清理台面）
- 实时监测玩家操作（拖拽、点击、顺序验证）
- 触发提示/纠正（非 LLM，先用规则）

#### 关键任务示例（可扩展）
| 任务名称 | 步骤数 | 核心技能 | 错误类型 |
|---------|--------|----------|----------|
| 煮面条 | 8 | 顺序、工具、安全 | 顺序错、水温错 |
| 三明治 | 6 | 组装、材料识别 | 材料错、遗漏 |
| 清理 | 5 | 收尾、卫生 | 工具放错、遗漏 |

#### 输出
- 原始事件日志（每步：时间戳、操作类型、成功/失败、用时）

### 2️⃣ 数据与指标层（闭环核心）

#### 功能
- 实时采集 → 结构化存储 → 指标计算

#### JSON Schema 示例
```json
{
  "session_id": "20260212_001",
  "task_name": "煮面条",
  "timestamp": "2026-02-12T19:00:00",
  "step_id": 3,
  "step_name": "烧水",
  "action": "click_pot",
  "success": true,
  "assist_needed": 1,
  "duration_ms": 4500
}
```

#### 核心指标（可直接用于论文统计）
| 指标名称 | 计算公式 | 临床含义 |
|---------|----------|----------|
| 独立完成率 | 独立步骤数 / 总步骤数 | 自主能力 |
| 平均步骤时长 | Σ用时 / 步骤数 | 执行效率 |
| 提示频率 | 提示次数 / 总步骤 | 依赖程度 |
| 错误率（分类） | 错误步骤 / 总步骤（按类型） | 问题模式 |

#### 输出
- Session 汇总指标 JSON
- 趋势数据（多 session 对比）

### 3️⃣ LLM-Agent 层（第二性：解释与报告）

#### 两个明确角色（非聊天机器人）

**角色1：任务解释 Agent**
- **输入**：当前步骤、玩家操作、错误标签
- **输出**：1–2 句固定模板解释
- **Prompt 模板示例**：
  ```
  你是厨房教练，用 1–2 句简单中文解释：
  当前步骤：[步骤名]
  玩家做了：[操作]
  结果：[成功/错误类型]
  解释："现在我们要[正确行为]，因为[原因]。你刚才[玩家行为]，下次试试[建议]。"
  ```
- **约束**：句长≤20字，避免抽象概念，用玩家熟悉词汇

**角色2：报告生成 Agent**
- **输入**：Session 指标汇总 JSON
- **输出**：结构化报告（家长/治疗师可读）
- **报告模板**：
  ```
  本次厨房任务：[任务名]
  完成情况：独立完成 [X/Y] 步，提示 [N] 次，总用时 [T] 分。
  进步点：步骤 [A] 时长减少 20%。
  关注点：错误集中在 [类型]，建议多练 [子技能]。
  ```

#### 技术实现
- **LLM**：Grok / GPT-4o-mini 等（成本低、可控）
- **框架**：LangChain（工具调用 + 模板）
- **安全**：输入过滤、输出校验（关键词黑名单）

                                           ↓

### 3️⃣ **核心架构模块**

#### **M1: Payload & Schema层**
- 职责：数据契约定义
- 技术：Pydantic V2 + Enum
- 特点：类型安全，前后端一致，支持JSON Schema验证

#### **M3: Agent.step() 流水线**
```
Observation → Reasoning → Reflex → Structured Response
```
- 协调者：`core/agent.py`
- 特点：清晰的责任链，异步处理

#### **M4: Reflex层 (小脑)**
- 文件：`service/adl_rules.py`
- 特点：纯函数，无LLM，确定性规则
- 规则：物理/安全/解剖学法则，实时保护

#### **M5: API/I-O层**
- 文件：`app.py`
- 特点：纯管道，无业务逻辑
- 端口：8001，支持CORS

#### **M6: LLM客户端**
- 文件：`service/llm_client.py`
- 特点：纯语言能力，可切换模型
- 提供商：DeepSeek API (OpenAI兼容接口)

#### **M7: Reasoning层 (咨询层)**
- 文件：`core/reasoning.py`
- 定位：LLM = 咨询顾问
- 输出：Proposal，非最终行动
- 集成：M8失败记忆 + M9语义规则

#### **M8: 失败记忆系统**
- 文件：`core/memory.py`
- 特点：记录执行失败，提供失败分析
- 存储：Episodic Memory，支持上次失败检索

#### **M9: 语义规则向量数据库**
- 文件：`service/vector_db.py`
- 技术：ChromaDB + 文本嵌入
- 功能：语义规则检索，上下文相关约束注入
- 调试：`debug_rule.py`查看所有规则

#### **M10: L2停滞检测系统 (看门狗)**
- 文件：`core/reasoning.py` (Watchdog类)
- 功能：
  - **思想巨人检测**：连续IDLE/THINK动作监控
  - **鬼打墙检测**：动作循环识别 (滑动窗口分析)
  - **自动干预**：检测到停滞时强制THINK动作
- 阈值：连续3次空转或5步内只有1-2种不同动作
